<objective>
Create the data layer infrastructure for offline write queuing: PendingOperationEntity with DAO, SyncWorker with exponential backoff and max-retry logic, WorkManager + Koin WorkerFactory integration, and database migration from v1 to v2.

Purpose: Establish the persistent write queue and background sync mechanism that all offline writes will flow through. Without this foundation, no offline mutations can be captured or replayed.

Output: PendingOperationEntity table in Room, PendingOperationDao with Flow-based pending count, SyncWorker that drains the queue sequentially with 3-retry exponential backoff, Koin-managed WorkerFactory, database v2 migration.
</objective>

    **1. Create SyncWorker (`data/src/main/java/com/wenubey/data/worker/SyncWorker.kt`):**
    - Extend `CoroutineWorker(appContext, params)`
    - Constructor-injected dependencies: `PendingOperationDao`, `DispatcherProvider`
    - NOTE: Do NOT inject repository dependencies yet — the worker only needs the DAO for now. Actual Firestore write dispatching per operation type will be wired in Phase 3+ when cart/profile repositories add offline-write support. For now, the worker structure and retry logic must be complete.
    - `doWork()` implementation:
      - `withContext(dispatcherProvider.io())` wrapping
      - Check `if (isStopped) return Result.failure()` at the start (per Pitfall 6: handle CANCELLED state)
      - Max retries = 3 (Claude's discretion). Check `if (runAttemptCount > 3)` — if exceeded, get the current pending operation and mark it FAILED with errorMessage "Max retries exceeded", then return `Result.failure()`
      - Call `pendingOperationDao.getNextPending()` — if null, queue is drained, return `Result.success()`
      - Update operation status to `IN_PROGRESS` with current timestamp (`java.time.Instant.now().toString()`)
      - TRY block: For now, create a `when` block on `OperationType.valueOf(operation.operationType)` with a placeholder `TODO("Wire ${operation.operationType} to repository in Phase 3+")` for each type. Wrap in `runCatching { OperationType.valueOf(operation.operationType) }` with fallback to mark unknown types as FAILED.
      - On success: `pendingOperationDao.deleteById(operation.id)`, then call `enqueue(applicationContext)` to process next item, return `Result.success()`
      - CATCH block: Log with Timber, call `pendingOperationDao.incrementRetryCount()`. If `retryCount + 1 >= 3`, mark FAILED with `errorMessage = e.message` and return `Result.failure()`. Otherwise return `Result.retry()`.
    - Companion object with:
      - `const val UNIQUE_WORK_NAME = "sync_pending_operations"`
      - `fun enqueue(context: Context)`: Builds `Constraints` with `setRequiredNetworkType(NetworkType.CONNECTED)`, creates `OneTimeWorkRequestBuilder<SyncWorker>()` with `.setBackoffCriteria(BackoffPolicy.EXPONENTIAL, Duration.ofSeconds(30))`, enqueues via `WorkManager.getInstance(context).enqueueUniqueWork(UNIQUE_WORK_NAME, ExistingWorkPolicy.REPLACE, workRequest)`
