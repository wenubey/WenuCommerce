**SyncManager.kt changes:**

1. Add ConnectivityObserver as a constructor parameter:
```kotlin
class SyncManager(
    private val firestore: FirebaseFirestore,
    private val productDao: ProductDao,
    private val categoryDao: CategoryDao,
    private val dispatcherProvider: DispatcherProvider,
    private val connectivityObserver: ConnectivityObserver,
)
```

2. Add import for `kotlinx.coroutines.flow.first` at top of file.

3. Change `emitOfflineWriteQueued()` from a synchronous function to a suspend function that checks connectivity before emitting:
```kotlin
suspend fun emitOfflineWriteQueued() {
    val isOnline = connectivityObserver.isOnline.first()
    if (!isOnline) {
        _syncEvents.tryEmit(SyncEvent.OfflineWriteQueued)
    }
}
```

This ensures the "Saved locally, will sync..." snackbar only appears when the device is actually offline. When online, the write is queued AND SyncWorker is enqueued in the same call (CartRepositoryImpl already calls `SyncWorker.enqueue(context)` before `emitOfflineWriteQueued()`), so the user gets no confusing message.

4. Add import for ConnectivityObserver:
```kotlin
import com.wenubey.data.connectivity.ConnectivityObserver
```

**DataModule.kt changes:**

Update the SyncManager Koin definition in `syncModule` to inject the 5th parameter (ConnectivityObserver):
```kotlin
single { SyncManager(get(), get(), get(), get(), get()) }
```

This works because ConnectivityObserver is already registered as a singleton in `connectivityModule` and `syncModule` is loaded after `connectivityModule` (verified in existing appModules order — `syncModule` is listed before `connectivityModule` in the module list but Koin resolves lazily, so the order of `single{}` registration does not matter for constructor injection; what matters is that ConnectivityObserver is registered before SyncManager is first resolved, which happens at app startup).

**CartRepositoryImpl.kt — NO changes needed.** The `addToCart()` method already calls `syncManager.emitOfflineWriteQueued()` inside a `withContext(dispatcherProvider.io())` block which is a suspend context, so calling a suspend `emitOfflineWriteQueued()` is valid. Similarly, `updateQuantity()` does NOT call `emitOfflineWriteQueued()` (only `addToCart` does), and `removeFromCart()` does not either — so only the one call site in `addToCart()` needs to work, and it already does since it's in a suspend lambda.
